# 2026-05-18 ŌĆö macOS PATH / PDF engine path fix (issue #1)

Fix for [issue #1](https://github.com/dsebastien/obsidian-book-exporter/issues/1): on macOS, Obsidian is launched as an Electron GUI app with a stripped `$PATH` (only `/usr/bin:/bin:/usr/sbin:/sbin`). Pandoc spawned by the plugin therefore couldn't find `typst`, `xelatex` or any other engine even when those were installed via Homebrew (`/opt/homebrew/bin`) or MacTeX (`/Library/TeX/texbin`) and worked fine from Terminal. The plugin only exposed `pandocPath` ŌĆö no knob for the PDF engine or for augmenting `$PATH`.

## Changes

- `src/utils/spawn-env.ts` (new) ŌĆö `buildSpawnEnv(extraPath)`: prepends user-provided directories to `$PATH` on the env handed to spawned children. Returns `process.env` unchanged when no extras are configured (zero cost in the common case). OS-portable via `path.delimiter` (`:` on POSIX, `;` on Windows).
- `src/app/types/plugin-settings.intf.ts` ŌĆö added `pdfEnginePath` (full path to the PDF engine binary, forwarded as `--pdf-engine=<path>`) and `extraPath` (directories prepended to spawn env's PATH). Both default to empty.
- `src/app/services/pandoc-runner.ts` ŌĆö pipes `buildSpawnEnv(settings.extraPath)` into the spawned pandoc process; replaces the inline engine-name with the result of `pickPdfEngineArg(engine, book, settings)`. The picker forwards the configured path only when its basename matches the selected engine (a typst path won't silently get forwarded to a xelatex export). Yields to `--pdf-engine` in `book_export.pandoc_extra_args` to keep the per-book override authoritative.
- `src/utils/binary-probe.ts` ŌĆö accepts a `BinaryProbeOptions` object (`timeoutMs`, `env`) so the preflight probe runs under the same env the export will. Backwards-incompatible signature change; only caller is `src/app/plugin.ts`.
- `src/app/plugin.ts` ŌĆö preflight check now (1) probes pandoc with the augmented env and (2) probes typst with `pdfEnginePath` when set (otherwise the bare name `typst`). Notice message updated to mention the new setting.
- `src/app/settings/settings-tab.ts` ŌĆö two new fields under "External tools": **PDF engine path** and **Extra PATH directories**.
- `src/utils/spawn-env.spec.ts` + `src/app/services/pandoc-runner.spec.ts` ŌĆö coverage for the new helpers (env composition, basename matching, per-book override precedence).
- `README.md` ŌĆö new "macOS: pandoc / typst / xelatex not found" troubleshooting section explaining the stripped-PATH problem and the three settings that fix it.
- `documentation/Configuration.md` ŌĆö new rows for `pdfEnginePath` and `extraPath`.

## Design notes

- Did **not** add hardcoded macOS defaults (e.g. auto-prepend `/opt/homebrew/bin` on darwin). Surprising behavior to inject without the user opting in, and only relevant on one OS ŌĆö `extraPath` is the explicit, cross-platform escape hatch.
- The picker matches on basename rather than always forwarding the configured path. Reason: a user who configures `/opt/homebrew/bin/typst` and then switches `defaultPdfEngine` to `xelatex` should not get `--pdf-engine=/opt/homebrew/bin/typst`. Better to fall back to `$PATH` lookup and surface a clearer "xelatex not found" error.
- Business rule #5 (Pandoc is the single external dependency) is unaffected ŌĆö no new dependency, just better path resolution for the engines pandoc itself calls.

## Verification

- `bun run validate` ŌĆö green (tsc, 43 tests pass, lint clean).
- `bun run build` ŌĆö green.
- **Manual verification needed** on macOS (Apple Silicon + Intel): set `pdfEnginePath` to `/opt/homebrew/bin/typst` (or `/Library/TeX/texbin/xelatex`) and `extraPath` to `/opt/homebrew/bin:/Library/TeX/texbin`, then run an export. The original reporter (issue #1) is on Apple Silicon with Homebrew typst + MacTeX xelatex and is the natural beta tester.
