# 2026-04-29 — Quality-of-life batch (separators, auto TOC depth, pre-flight check, embeds, image width, front matter, numbering, citations, dup detection)

The previous evening's session left a list of nine open improvement requests. This entry tracks them as they ship — one clean commit per item.

## C1 — Inlined-note separator setting

User feedback: "yes, add a setting to enable/configure/disable separators between inlined notes."

When a manifest section bundles several notes, the compiler used to flow them straight into one another. Without a visual cue, readers couldn't tell where one atomic note ended and the next began — and the user had previously misread this as recursive inlining.

Added a new plugin setting **Inlined-note separator** with four modes:

- `none` — nothing emitted (default; preserves the previous behaviour for existing books).
- `rule` — a Pandoc thematic-break glyph row (`* * *`). Distinct from the `---` syntax the compiler reserves for manual page breaks.
- `blank` — an extra blank line on top of the spacing between blocks.
- `subheading` — emit each note's display title as a heading one level below the section heading (capped at H6).

Per-book override: `book_export.inlined_note_separator: <mode>`.

### Implementation

- New `InlinedNoteSeparator` type in `src/app/domain/book-manifest.intf.ts`.
- New `inlinedNoteSeparator: InlinedNoteSeparator` on `PluginSettings` (default `none`).
- New `inlinedNoteSeparator?` on `BookExportOverrides` + parsed in `extractOverrides` via a new `isInlinedNoteSeparator` guard.
- `ManuscriptCompiler.renderSection` rewritten so it tracks "is this the first emitted note?" and inserts the separator between successive non-empty notes only. The `subheading` mode emits the heading line itself; the other modes use a small `renderNoteSeparator()` helper that returns `'* * *'`, `''` (extra blank), or `null` (no extra line).
- Settings tab: new dropdown under **Note processing** with a human-readable label per mode.
- Docs updated: `docs/configuration.md`, `docs/release-notes.md`, `documentation/Domain Model.md`.

## C2 — Auto TOC depth from heading levels actually present

User feedback: "implement 3, computing it from the levels actually present"

The static `tocDepthDefault: 2` was a guess; many books don't fit it (parts + chapters need 3, deep textbooks need 4). Made the compiler derive the value from the manifest itself.

### Implementation

- `BookParser.parse` now also returns `maxHeadingLevel: number` on `ParsedBook`, computed by walking the section tree and tracking the deepest `level`.
- New plugin setting `tocDepthAuto: boolean` (default `true`). When on, `pandoc-runner` picks `--toc-depth` from `book.maxHeadingLevel`. The static `tocDepthDefault` is now only the fallback (used when the setting is off, or when the manifest has no parseable heading).
- `pickTocDepth()` helper in `pandoc-runner.ts` encapsulates the resolution order: per-book `book_export.toc_depth` override → auto from manifest → static default. Result is clamped to `[1, 6]`.
- Settings tab: new toggle "Auto TOC depth"; the static field is renamed "TOC depth (fallback)" so the relationship is obvious.
- Docs: `docs/configuration.md` (settings table), `docs/release-notes.md`.

## C3 — Pre-flight binary check on plugin load

Until now, a missing `pandoc` or `typst` only surfaced when the user invoked an export command and saw a cryptic `child_process spawn ENOENT` notice. Promoted this to a load-time check so the user finds out immediately.

### Implementation

- New helper `src/utils/binary-probe.ts` exposes `probeBinary(bin, timeoutMs?)`. Spawns `<bin> --version`, returns `{ ok, versionLine?, error? }`. Resolves (never rejects) on missing binaries; self-times out after 5s to keep `onload` from stalling.
- `BookExporterPlugin.onload` fires `runPreflightCheck()` (fire-and-forget). It probes `pandoc` and — when the configured PDF engine is `typst` — also probes `typst`. LaTeX engines (xelatex/tectonic/lualatex) are NOT probed: their `--version` is environment-sensitive and would produce false negatives. Pandoc itself surfaces those errors at export time.
- Failures show a 10s `Notice` with the actionable next step ("Set Settings → Book Exporter → Pandoc path", "Install from typst.app, or pick a different PDF engine"). Successes log to console only — no UI noise on a healthy install.

## C4 — Opt-in `![[Note]]` embed expansion

Until now, note embeds inside inlined notes were silently dropped (only image embeds survived). For atomic-note authors who routinely use `![[Foo]]` to compose a longer piece, that meant losing content in the export. Added an opt-in expansion path.

### Implementation

- New `inlineNoteEmbeds: boolean` (default `false`) and `noteEmbedMaxDepth: number` (default `3`) on `PluginSettings`.
- `BodyTransformer` now takes an `opts` object (`expandNoteEmbeds`, `noteEmbedMaxDepth`, `sectionsToSkip`) and threads `(depth, visited)` through `transform()` and `rewriteImages()`.
- New `expandNoteEmbed()` helper:
  1. Strips `#section` / `^block` anchors and resolves the linkpath via `getFirstLinkpathDest`.
  2. Skips non-Markdown targets (returns `null` so the caller drops the embed as before).
  3. Returns the alias / basename when `depth + 1 >= noteEmbedMaxDepth` or when the target is already in the `visited` set (cycle).
  4. Reads the file, strips frontmatter, applies `sections_to_skip`, drops the leading H1, then recurses `transform()` with `depth + 1` and `visited ∪ {file.path}`.
- Settings tab: new toggle "Inline note embeds" + numeric "Note embed max depth", placed at the top of the **Note processing** section so the relationship with the existing "Sections to skip" is obvious.
- Docs: `docs/configuration.md`, `docs/release-notes.md`, `documentation/Domain Model.md`.

## C5 — Image max-width for Typst

Over-sized images (e.g. 4K screenshots dropped into a vault note) used to bleed past the page text frame in Typst PDFs because the engine's default is to size images at their native pixel dimensions. Added a configurable directive injected in the Typst preamble.

### Implementation

- New `typstImageWidth: string` setting (default `100%`). Accepts any Typst length (`100%`, `80%`, `15cm`); empty disables the directive entirely.
- The Typst preamble is now built dynamically by `buildTypstPreamble(settings)` — emits `#set image(width: <value>)` when the setting is non-empty, then the existing `quote` block styling. Replaces the previous `TYPST_PREAMBLE` constant.
- Settings tab: new "Typst image width" text input under **Rendering**.

## C6 — Front matter vs body matter

Convention for typeset books: front-matter pages (foreword, preface, dedication, table of contents) are numbered with lowercase roman numerals; the body matter restarts at `1`. Added per-book support for this without giving up the heading-driven manifest contract.

### Implementation

- New `BookExportOverrides.frontMatterSections?: string[]` on the domain type. Parsed from `book_export.front_matter_sections` in the manifest frontmatter.
- The compiler tracks two transitions:
  1. **Manuscript open** — when any top-level section is flagged as front matter, emit `FRONT_MATTER_OPEN` (Typst: `#set page(numbering: "i")`, LaTeX: `\frontmatter`) right before the title.
  2. **Front → body switch** — when iterating the top-level sections, the first one whose normalised title is NOT in the front-matter set triggers `BODY_MATTER_OPEN` (Typst: `#pagebreak(to: "odd")` + counter reset + `#set page(numbering: "1")`; LaTeX: `\mainmatter`). The transition includes its own page break, so the chapter/part break that would normally precede this section is suppressed.
- Title matching is case-insensitive (`normaliseTitles()`).
- HTML/EPUB ignore the raw blocks — page numbering doesn't apply to reflowable formats anyway.

## C7 — Section numbering setting

Pandoc's `--number-sections` flag was previously only reachable through `pandoc_extra_args`. Promoted it to a first-class setting so users don't have to know the CLI shape.

### Implementation

- New `numberSections: boolean` (default `false`) on `PluginSettings`.
- New `numberSections?: boolean` on `BookExportOverrides`, parsed from `book_export.number_sections`.
- `pandoc-runner.buildArgs` reads `book.overrides.numberSections ?? settings.numberSections` and pushes `--number-sections` when truthy.
- Settings UI: new toggle under **Rendering**.
- Docs: `docs/configuration.md` (settings + per-book table), `docs/release-notes.md`.

## C8 — Citation rendering via `pandoc-citeproc`

Citations were entirely missing from the export pipeline; users with academic-leaning notes had no way to drop a `[@smith2020]` and get a rendered reference.

### Implementation

- Two new optional fields on `BookMetadata`: `bibliographyPath` and `cslPath`. Both store **absolute** filesystem paths once resolved.
- `BookParser` extracts `bibliography:` and `csl:` from the manifest's frontmatter via a new generic helper `resolveLocalPath()` (modeled after `resolveCover` but URL-free — citation resources must be local files for pandoc-citeproc to read).
- `buildMetadataYaml` writes both paths into `metadata.yaml` so pandoc-citeproc resolves them at export time.
- `pandoc-runner.buildArgs` pushes `--citeproc` whenever `book.metadata.bibliographyPath` is defined. CSL is consumed by pandoc through the YAML field; no additional flag is needed.
- Docs: `docs/usage.md` (frontmatter table), `docs/release-notes.md`, `documentation/Domain Model.md`.
