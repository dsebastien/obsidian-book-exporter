## C1 — Inlined-note separator setting

User feedback: "yes, add a setting to enable/configure/disable separators between inlined notes."

When a manifest section bundles several notes, the compiler used to flow them straight into one another. Without a visual cue, readers couldn't tell where one atomic note ended and the next began — and the user had previously misread this as recursive inlining.

Added a new plugin setting **Inlined-note separator** with four modes:

- `none` — nothing emitted (default; preserves the previous behaviour for existing books).
- `rule` — a Pandoc thematic-break glyph row (`* * *`). Distinct from the `---` syntax the compiler reserves for manual page breaks.
- `blank` — an extra blank line on top of the spacing between blocks.
- `subheading` — emit each note's display title as a heading one level below the section heading (capped at H6).

Per-book override: `book_export.inlined_note_separator: <mode>`.

### Implementation

- New `InlinedNoteSeparator` type in `src/app/domain/book-manifest.intf.ts`.
- New `inlinedNoteSeparator: InlinedNoteSeparator` on `PluginSettings` (default `none`).
- New `inlinedNoteSeparator?` on `BookExportOverrides` + parsed in `extractOverrides` via a new `isInlinedNoteSeparator` guard.
- `ManuscriptCompiler.renderSection` rewritten so it tracks "is this the first emitted note?" and inserts the separator between successive non-empty notes only. The `subheading` mode emits the heading line itself; the other modes use a small `renderNoteSeparator()` helper that returns `'* * *'`, `''` (extra blank), or `null` (no extra line).
- Settings tab: new dropdown under **Note processing** with a human-readable label per mode.
- Docs updated: `docs/configuration.md`, `docs/release-notes.md`, `documentation/Domain Model.md`.
