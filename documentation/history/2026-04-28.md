# 2026-04-28 — Plugin scaffolded, MOBI dropped, Typst as default PDF engine, manifest contract redesigned

## Initial scaffold (morning)

Bootstrapped `obsidian-book-exporter` from `obsidian-plugin-template`. Removed inherited template state (CHANGELOG, versions.json, manifest, package.json identity, `init` script and `TEMPLATE_USAGE.md`).

## Decisions

- Desktop-only plugin (`isDesktopOnly: true`) — needs `pandoc` over `child_process`.
- TOC parsing uses level-2 headings (`Front Matter`, `Chapters`, `Back Matter`) + bulleted wikilinks. Configurable.
- Compiler strips frontmatter, drops the first H1 of each note, demotes remaining headings, rewrites Obsidian-only syntax (callouts, embeds, comments).
- Pandoc is fed metadata via a generated YAML file rather than CLI flags to avoid escaping issues.
- Settings stay immutable — Immer `updateSettings(draft => ...)`.
- Any Markdown note can be a manifest; no specific tag is required.

## Afternoon — MOBI removed, PDF defaults switched

- **Dropped MOBI entirely.** Calibre `ebook-convert` removed. Modern Kindles and KDP accept EPUB; the install footprint of Calibre wasn't justified.
- **Pandoc stays a hard prereq.** Investigated alternatives: `pandoc-wasm` (experimental, ~50MB bundle, no LaTeX integration), bundling the binary (multi-platform, ~150MB×N — not viable), pure-JS EPUB + Electron `printToPDF` for PDF (browser-grade page breaks — not professional enough for books). Conclusion: keep Pandoc, document the install clearly.
- **Default PDF engine: `typst`.** Single small binary, modern, fast, professional output. Replaces `xelatex` as the default (LaTeX engines remain available as alternatives). Removes the LaTeX install hurdle for new users.
- Removed `calibre-runner.ts`, `ExportFormat = 'epub' | 'pdf'`, `defaultFormats: [epub, pdf]`, settings entry for ebook-convert path, and the MOBI command.

## Evening — Manifest contract redesigned (v2)

The original "Front Matter / Chapters / Back Matter" body model was too opinionated. It forced a single structural shape and didn't fit real books with parts, subsections, or non-standard sectioning (foreword, acknowledgements, multiple appendices). Replaced with a **heading-driven, structure-agnostic** contract.

### New contract

- `# H1` is the book title (frontmatter `title` wins; basename is the last fallback).
- Every `## H2` … `###### H6` is a section at the matching level. Sections nest under the previous higher-level section.
- Every bullet under a section that contains one or more `[[wikilinks]]` contributes those links — in source order — to the section.
- No reserved heading names. No required structural shape. The plugin walks whatever heading tree is there.

### Cleanup added

- New plugin-level + per-book setting **`sectionsToSkip` / `sections_to_skip`** (default `Related, References, Title Options, Target Audience`). Applied at two points:
  - **Parse time, on the manifest body** — drops authoring scaffolding (working notes, target-audience notes, references, related-notes index) so it stays in the manifest but never reaches the export.
  - **Compile time, on each linked note** — drops housekeeping sections from atomic notes.
  Same logic in both places: case-insensitive heading match, fence-aware, removes heading + body until a same-or-higher heading.
- Shared helpers extracted to `src/utils/markdown.ts` (`stripFrontmatter`, `stripSkippedSections`) — eliminates the parser/compiler circular import that would have arisen from sharing the function.

### Changes

- `BookSection` (recursive `{ level, title, notes, children }`) replaces the old `frontMatter / chapters / backMatter` triple.
- `NoteReference` formalised (`filePath`, `displayTitle`).
- `BookParser.parseBody` rewritten as a heading-stack walk. No more bucket-based parsing; no more reserved heading lookups.
- `ManuscriptCompiler` rewritten as a recursive section renderer. Heading demotion offset is `parentLevel - 1`. New `stripSkippedSections` helper.
- Page break logic now keys off the **lowest-numbered heading level present in the manifest** — so a parts-and-chapters book breaks per part, a flat book breaks per chapter, automatically.
- `BookValidator` traverses the new tree and reports unresolved links with breadcrumb paths.
- Settings tab: removed front/chapters/back-matter heading inputs; added a **Sections to skip** textarea.
- Removed `frontMatterHeading`, `chaptersHeading`, `backMatterHeading` from settings.
- Documentation surfaces (`README.md`, `docs/`, `documentation/`, vault note) updated to the new contract.

### Why

User feedback (verbatim): "the plugin is wrong. The way I want it to work: A book note used with this plugin has an h1: title of the book, h2s: chapters (or parts), optionally h3s, and lists of links pointing to notes whose content should be added to the book. As the content is regrouped, it should be cleaned up so that the structure remains fully coherent. The plugin needs to support configuration for sections to skip in processed notes."

Real books don't fit a fixed three-bucket model. The user's existing manifest (`The Context Layer (Book).md`) uses `## Part I → ### Chapter 1` — would have been impossible under the old contract. The new contract handles arbitrary structures while keeping the inlining cleanup that makes published output coherent.

## Late evening — output and temp paths leave the vault

User feedback: "nok for the plugin to store data in the plugin folder while exporting. Temp files should be in the system's temp folder (in a clearly named folder). Also, the final files should be put in a configured folder (eg downloads). If not configured yet, then the plugin should refused to export."

Two changes that take the plugin out of the vault entirely:

- **Temp files now live in `os.tmpdir()`.** Each export creates a fresh per-book directory via `fs.mkdtemp(path.join(os.tmpdir(), 'obsidian-book-exporter-<slug>-'))`. No more `<vault>/.obsidian/plugins/obsidian-book-exporter/.tmp/`. Cleaned up after every export unless `keepTempFiles` is set.
- **Output is an absolute filesystem path now**, not vault-relative. Default is empty; the plugin refuses to export with a clear Notice ("Set Default output folder in Settings → Book Exporter, e.g. ~/Downloads") until configured. Supports `~` expansion. Per-book `book_export.output_dir` follows the same rules. The vault adapter's `getFullPath` is no longer used in the export pipeline — output paths are now plain absolute paths.
- Added an `expandHome()` helper exported from `exporter.ts`. Used by both the export pipeline and the **Open exports folder** command.
- Business rules updated: rule 3 now mandates absolute filesystem paths for output (was vault-aware path resolution); rule 4 now mandates OS temp directory for scratch space (was the plugin tmp dir).

Verified end-to-end on `The Context Layer (Book)`: set `defaultOutputDir: "~/Downloads"` in `data.json`, ran EPUB export, file landed at `/home/dsebastien/Downloads/the-context-layer_2026-04-28.epub`. No `.tmp` directory was created in the plugin folder. The OS temp directory was cleaned automatically after the export.

## Late evening — defaultAuthors plugin setting

Author resolution now has a third tier: frontmatter `authors:` → plugin `defaultAuthors` setting → `["Anonymous"]`. Avoids having to repeat the author across every book manifest. Per-book frontmatter still wins.

Verified end-to-end on `The Context Layer (Book)`: deployed plugin, enabled via `obsidian plugin:enable`, set `defaultAuthors: ["Sébastien Dubois"]` in `data.json`, ran `preview-manuscript` → `metadata.yaml` shows `author: "Sébastien Dubois"`. EPUB export produced 4.4 MB book at `Exports/Books/the-context-layer_2026-04-28.epub`.
